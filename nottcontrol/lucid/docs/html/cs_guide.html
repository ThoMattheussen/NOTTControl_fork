Arena SDK: C# Programmer’s Guide

Arena SDK: C# Programmer’s Guide#

This guide outlines how to create applications using the C# API in the Arena SDK. See the Arena SDK Overview for more general information. This document is intended for developers creating applications for LUCID cameras.

Namespaces#

The following namespaces are relevant to C# applications created with the Arena SDK:

  • GenTL: GenTL is a low-level API that defines a standard interface to a camera. It defines five layers of abstraction for a camera system: System, Interface, Device, Stream, and Buffer. LUCID has implemented GenTL modules for System, Interface, Device, and Stream. This means that you can configure settings for these modules through the SDK and through ArenaView.

  • GenAPI: GenAPI is an even lower-level API that defines how requests from an application are translated into specific instructions performed on the camera’s register. It is defined by the GenICam standard.

  • ArenaNET: The C# library of the Arena SDK. The source files for this namespace are located in the C:\ProgramData\Lucid Vision Labs\ArenaNET folder.

  • SaveNET: A LUCID library used for saving images in a desired format. The source files for this namespace are located in the C:\ProgramData\Lucid Vision Labs\SaveNET folder.

GenTL layers#

To help reduce complexity, the GenTL standard has a layered structure for libraries implementing the standard. Each layer, referred to as a “GenTL layer”, abstracts specific aspects of the communication and data handling between a software application and the camera.

In the Arena SDK implementation, the GenTL layers for System, Interface, Device, and Stream have a corresponding GenTL module, but the Buffer layer does not. (GenTL parameters related to the Buffer layer are available in the Stream node map.)

Getting an image file#

To obtain an image file via the SDK, you need to open the system and create a device; create node maps and use the nodes; acquire an image; and save the image.

Instantiating the system object#

Instantiate the system object (⭐1) with the OpenSystem() global function. The system object is the entry point into the Arena C# SDK. Next, enumerate the devices using the UpdateDevices() function of ISystem, which is the interface to the system object. You can then use this vector to create a device using the CreateDevice() function (⭐2).

What this looks like in code:

// initializing example 
ArenaNET.ISystem system = ArenaNET.Arena.OpenSystem();
system.UpdateDevices(100);
if (system.Devices.Count == 0)
{
        Console.WriteLine("\nNo camera connected\nPress enter to complete");
        Console.Read();
        return;
}
ArenaNET.IDeviceInfo selectedDeviceInfo = SelectDevice(system.Devices);
ArenaNET.IDevice device = system.CreateDevice(selectedDeviceInfo);

Creating node maps and using nodes#

Node maps are collections of nodes. A node is a feature or command related to a camera function. You use nodes to get and configure camera settings, and to initiate commands. There are two types of node maps: the main node map, which is obtained directly from the XML file on the camera, and the GenTL node maps, which access features through the software. The main node map is the most commonly used node map.

Through these node maps, you get, set, or activate the desired nodes. Use the GetNode() method (⭐5) of the INodeMap interface to retrieve the node, and the FromString() method of the IValue interface to set the node value.

Examples of getting/setting node values:

// Getting the acquisition mode
var acquisitionModeNode = (ArenaNET.IEnumeration)device.NodeMap.GetNode("AcquisitionMode");
String acquisitionModeInitial = acquisitionModeNode.Entry.Symbolic;

// Setting the acquisition mode
acquisitionModeNode.FromString("Continuous");

The firmware documentation on the camera contains information on the nodes used for that particular camera.

To access the firmware documentation from ArenaView:

  1. In ArenaView, ensure that the Start pane is open.

  2. Under the desired camera, select “Documentation”. The documentation opens in your browser.

Node types#

Each node described above has a defined type (e.g., IString, IFloat, IEnumeration). These types are defined in the GenICam standard.

Acquiring an image#

The first step to acquiring an image is to run StartStream(), which starts the internal streaming engine.

Next, there are several ways to obtain an image from the camera. You can use the GetBuffer() (⭐6) method of IDevice, which is the interface to a device object, to create a buffer object from the streaming engine. IBuffer is the interface to this buffer object. You can then cast the buffer object as an image object (⭐7).

Alternatively, you can also use the GetImage() method, which combines the two steps above.

Examples:

  • Acquisition, Introduction

  • Acquisition, RDMA (Atlas10 only)

  • Acquisition, TCP (Atlas10 only)

Obtaining an image file#

Once an image object has been created, it typically needs to be saved as a file. From the IImage interface (⭐7, ⭐8), you can obtain an image file using the SaveNET library, or obtain as a Bitmap using classes in the System.Drawing namespace in .NET.

SaveNET

To obtain an image object as an image data, you use the DataArray class of IBuffer. You can then save the image data as a file using the SaveNET library’s ImageWriter class.

System.Drawing

Alternatively, you can use .NET’s System.Drawing.Bitmap class to obtain the image data, and then use the System.Drawing.Image.Save() function to obtain an image file.

Working with the Arena SDK#

This section shows useful things your application can do with the SDK. Some features are for specific camera models or camera families.

Callbacks#

Note

For information on callbacks as they are used with events, see Events.

When registering a callback, you must specify a specific event cause that will trigger the callback function in addition to the callback function itself.

// register callback
// create delegate for registering callback
ArenaNET.INode.del d = new ArenaNET.INode.del(PrintIntegerNodeValue);
IntPtr callback = eventTestTimestamp.RegisterCallback(d);

In the example Callbacks, On Node Change, the callback is invoked when the node value for the height is changed/invalidated because the payload size depends the height.

// Modify Height to invoke callback on PayloadSize
//    The value of PayloadSize depends on several other nodes.
//    including Height. Therefore, changing the value of Height
//    changes the value of and invalidates PayloadSize, which then
//    invokes the callback.
Console.WriteLine("{0}Change height once", TAB2);

heightNode.Value = HEIGHT_ONE;


// Manually invalidate PayloadSize for callback
//    Apart from changing the value of a node, nodes can be
//    invalidated manually by calling InvalidateNode. This also
//    invokes the callback.
Console.WriteLine("{0}Invalidate PayloadSize", TAB2);

payloadSizeNode.InvalidateNode();

When the callback is invoked, the program flow goes to the callback function below. Once the program flow goes through the function, the it returns to the point where the callback function was triggered.

Once you no longer require the callback, be sure to use the DeregisterCallback() function to prevent memory leaks.

The Arena SDK documentation has several examples of invoking callback functions.

Examples:

  • Callbacks, On Node Change (described above)

  • Callbacks, Polling

  • Callbacks, Image Callbacks

  • Callbacks, Multithreaded Image Callbacks

Chunks#

A chunk is a fragment of metadata appended to raw images. Examples of chunks that may be available on your camera include the following: CRC, histogram stats, timestamp, line status, sequencer set, pixel format, width/height, offset X/Y, pixel dynamic range min/max, gain, black level, exposure time.

Note

Q: You can get the exposure time from the device. Why would you use chunk data instead?

A: The chunk data is particular to the image. If the camera has autoexposure control enabled and you run the camera at a high frame rate, the exposure time changes rapidly. As a result, there may be a discrepancy in the exposure time that you determine through the device and the actual exposure time for a given image. Under these conditions, it is better to use the chunk data.

Examples:

  • Chunk Data, Introduction

  • Chunk Data, CRC Validation

Events#

LUCID cameras can deliver events to inform the host system of the current operation of the camera. For instance, an event can be generated when the exposure starts, which is then sent to the host before the end of exposure. Events are asynchronous with the image acquisition and data transfer and can be sent at any time by the camera.

Defined by the GenICam standard, these events from the camera are generated and processed using the sequence below, which is from the example Callbacks, On Event.

  1. Get the event node.

  2. Initialize the events engine.

  3. Register the callback using the event node. This step is similar to the example covered in the callbacks section.

  4. The camera generates an event.

  5. The program flow comes to the WaitOnEvent() function. The documentation for this function contains more details on event processing. It is only at this function that the event is processed, and a callback is invoked.

  6. Deregister the callback and deinitialize the event to prevent memory leaks.

Example:

  • Callbacks, On Event

Streamables files#

A camera feature marked as streamable can have its current value stored to and loaded from a file, called a streamables file. Streamables files can be downloaded from the camera and uploaded to the camera (or another camera) using ArenaView or the Arena SDK. Like user sets, streamables offer a method of saving a camera’s settings, but unlike user sets, streamable files can be used to apply settings to another camera.

Note

The term “streamable” means that the feature can be streamed to a file. It is unrelated to video streaming.

Example:

  • Streamables

Triggering#

The most common example of triggering a camera is when a software event or hardware signal initiates the acquisition of a single image, which is what happens when the trigger selector is set to Frame Start.

Other trigger selectors are shown below.

Trigger selector

Description of initiated action

Acquisition Start

Capture images at the free running frame rate.

Frame Start

Capture a single image.

Frame Burst Start

Capture a configurable number of images.

Exposure Active

Expose the image sensor for as long as there is a high signal.

Frame Burst Active

Triggers images for as long as there is a high signal.

For more information, see the on-camera documentation under Device Nodes > Acquisition Control.

Although the hardware trigger to a camera does not come in through the API, there are related settings that can be configured through the C# API. Examples of this for the GPIO include LineMode and LineSelector. For more information on GPIO, see the related App Note on our website.

Handling exceptions#

You can use a try-catch block in the main function of the code. The excerpt below shows different catch statements for the GenICam namespace and the standard namespace.

try
{
	// Attempt to run code that may produce exceptions
}

catch (ArenaNET.TimeoutException ex)
{
	Console.WriteLine( "GenICam exception thrown: ", ex);
}

catch (Exception ex)
{
	Console.WriteLine("Exception thrown: {0}", ex);
}

Building applications with the Arena C# API#

The easiest way to start exploring how the API works is to first ensure that the code samples included with the Arena SDK installation compiles and executes correctly on your system. Next, take an existing code snippet and modify it to suit your needs.

Windows

Ensure that you have configured your Windows system to use the Arena SDK. The Windows installation includes example files and settings files for Visual Studio 2015 and Visual Studio 2019.

The code samples available in this documentation are also included in the Arena SDK installation. The default location for the installed code samples is shown below.

C:\ProgramData\Lucid Vision Labs\Examples\src\C# Source Code Examples

To open the solution of C# code examples, open the appropriate solution file in Visual Studio 2015.

  • “CS examples.sln” for Visual Studio 2015

  • “Cs Examples_vs2019.sln” for Visual Studio 2019

Creating a standalone solution in Visual Studio 2015

When creating a standalone solution in VS2015, be sure to add the SaveNET.dll and ArenaNET_v140.dll files as references. You can find these files in the appropriate folder under the “C# Source Code Examples/OutputDirectory/Windows”. You will need to build each file in the solution of C# code examples mentioned above.

In your standalone solution, ensure that you have defined the Configuration (i.e., Release or Debug) and Platform (x64).

Tip

Turn off the screensaver and auto-sleep on your host to prevent unexpected results from your applications, especially those used in production.
